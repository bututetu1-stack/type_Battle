# オンライン・タイピングバトルロワイヤル 仕様書 v0.2

**コードネーム**：（仮）TypeRoyale
**コンセプト**：テトリス99型の「つぶし合い」をタイピングで実現するオンライン対戦ゲーム。
**技術前提**：GitHub Pages（静的ホスティング）＋ Firebase Realtime Database。Unity不使用。

---

## 1. コンセプト

全プレイヤーが同時にタイピング課題を打ち続け、速く正確に打つほど相手に「おじゃま単語」を送り込む。送られた側はバックログ（処理待ち単語の山）が膨らみ、上限を超えると脱落。最後の1人が勝利する。

テトリス99との対応関係：

| テトリス99 | 本作 |
|---|---|
| 盤面（フィールド） | バックログ（縦に積まれた処理待ち単語） |
| ブロックを設置してライン消去 | 単語をノーミスで打ち切る |
| ライン消去でおじゃま送信 | 連鎖・速度でおじゃま単語を送信 |
| おじゃまブロックがせり上がる | おじゃま単語がバックログ末尾に挿入される |
| トップアウト（積み上がって死) | バックログ上限オーバーで脱落 |
| 火力＝攻撃と防御を兼ねる | タイプ速度＝攻撃と防御を兼ねる |

## 2. 基本ルール

### 2.1 勝利条件
- 最後まで生き残った1人が勝利（バトルロワイヤル）。
- 脱落順に下位から順位が確定する。

### 2.2 バックログ（＝盤面）
- 各プレイヤーは縦に積まれた「処理待ち単語のスタック」を持つ。
- **先頭の単語**が現在打つべき対象。打ち切ると先頭が消え、次が繰り上がる。
- 新規単語は**末尾に追加**される。供給源は2つ：
  1. **ベース供給**：一定間隔で自動的に1単語追加（後述の加速あり）。
  2. **おじゃま**：他プレイヤーからの攻撃で挿入される。
- バックログ最大数を超える追加が発生した瞬間 → **トップアウト（脱落）**。

### 2.3 ベース供給と加速
- ゲーム開始時は緩やか（例：4.0秒ごとに1単語）。
- 時間経過で供給間隔が短縮（例：30秒ごとに×0.85）。
- これがテトリスのレベル上昇＝落下速度アップに相当し、**ゲームが必ず終局へ向かう圧**を作る。打たずに耐える戦法を封じる役割も持つ。

### 2.4 単語のクリアとミス
- 単語を**1文字もミスせず**打ち切る → 連鎖（コンボ）+1。
- 単語の途中で1回でもミス入力 → その単語は連鎖に加算されず、**連鎖リセット**。
- ミスしても単語自体は打ち切れば消える（詰まって自滅はしない）。あくまで火力が出なくなるのがペナルティ。

## 3. 攻撃システム（核心）

### 3.1 攻撃の発生
攻撃は「単語1個ごと」ではなく、**連鎖の節目**で発生させる（毎単語送信だと単調なため）。

**連鎖マイルストーン攻撃**（暫定値・要調整）

| 到達連鎖数 | その瞬間に送るおじゃま |
|---|---|
| 5連鎖 | 1個 |
| 10連鎖 | 2個 |
| 15連鎖 | 3個 |
| 20連鎖 | 4個 |
| 以降5連鎖ごと | +1個 |

**加算ボーナス**
- **速度ボーナス**：規定の打鍵速度（KPM）を超えている間、マイルストーン攻撃を×1.5。
- **難単語ボーナス**：規定文字数以上の単語をノーミスで打ち切ると +1個。
- **パーフェクト継続**：直近N単語連続ノーミス時に倍率（テトリスのB2B相当）。

> 数値はすべて v0.1 のたたき台。実プレイでのバランス調整前提（→ §11）。

### 3.2 おじゃま単語（garbage）
- 受信したおじゃまはバックログ末尾に挿入される。
- 通常単語と視覚的に区別（例：赤系色・記号混じり・やや長め）。
- **おじゃま単語をクリアしても攻撃ポイントは発生しない**（おじゃまで火力を稼げない＝テトリスのおじゃま消しと同様の非対称性）。ただし連鎖カウントには加算する。

### 3.3 相殺システム（テレグラフ＋キャンセル）
テトリスの「おじゃま予告」を再現する、深さの源。

1. 攻撃を受けると、即座には挿入されず**画面横の「予告ゲージ」**に溜まる。
2. 一定の遅延（例：1.5秒）後に確定し、バックログに挿入される。
3. 確定前に自分が攻撃を発生させると、その攻撃で予告ゲージを **1:1で相殺**。
4. 相殺しきれなかった分のみ自分に確定挿入。逆に自分の攻撃が余れば、その余剰がターゲットへ飛ぶ。

→ 「受けてから打ち返して帳消しにする」攻防が生まれる。

### 3.4 ターゲティング（4モード）
テトリス99の4モードをそのまま移植。ボタン/キーで切替。

- **ランダム**：生存者から無作為。
- **反撃**：直近で自分を攻撃してきた相手を狙う。
- **とどめ**：バックログが多い＝瀕死の相手を狙う。
- **強敵**：バッジ保有数が多い相手を狙う（ハイリスク・ハイリターン）。

### 3.5 バッジ / KOボーナス
- 自分の攻撃でトドメを刺した相手の脱落をKOとしてクレジット。
- KO数に応じてバッジ獲得 → 自分の総攻撃力に倍率。
  - 1バッジ +25% / 2バッジ +50% / 3バッジ +75% / 4バッジ +100%（上限・要調整）。
- 強い相手を倒すほど雪だるま式に強くなる、テトリス99の終盤の緊張感を再現。

## 3.6 逆転・バランス設計（速度だけで勝たせない）

**設計思想**：勝敗を「速度」一軸で決めない。速度・正確性・粘り・立ち回り・運の複数軸を用意し、タイピングが遅いプレイヤーにも複数の勝ち筋を残す。これがバトロワを「速い人が必ず勝つ作業」から「誰にでも勝ち筋がある対戦」へ変える。

**(A) 正確性が火力の主軸**
- 火力は連鎖（ノーミス連続）依存。速くてもミスを連発すると連鎖が切れ続け、火力ゼロになる。
- 遅くても正確な人は長い連鎖を維持して安定火力を出せる。→「正確に打てること」自体が強さになる。

**(B) 火事場の馬鹿力（ピンチ時攻撃倍率）**
- 自分のバックログが危険域（例：上限の70%以上）にある間、攻撃倍率アップ（例：×1.5〜×2）。
- ボコられて瀕死の人ほど一撃が重くなり、一発逆転の起点になる。
- テトリス99で集中砲火された人が耐えて打ち返す展開を再現。

**(C) 強者ほど狙われる構造（独走の抑制）**
- バッジ（KO数）が多い＝強者に「強敵ターゲット」モードの攻撃が集中。
- KOを稼ぐ速い人は自然と全体の的になり、火力過多でも袋叩きにされる。
- → 速い人が無限に勝ち続けられない、エメルジェントなバランス（テトリス99の肝）。

**(D) お宝単語＆アイテム（ラバーバンド補正）**
- バックログにランダムで「お宝単語」が出現。クリアするとアイテムを抽選獲得。
- アイテム例：シールド（次の被弾無効）／おじゃま一掃（自分のおじゃまを消す）／反射（次の被弾を相手へ跳ね返す）／全体攻撃（全生存者に少量）／ブレーキ（自分のベース供給を一時停止）。
- **下位（順位が低い／バックログが多い）プレイヤーほど良いアイテムが出やすい**ラバーバンド補正（マリオカート方式）。速さに依存しない逆転変数を注入する。

**(E) 粘りボーナス（任意）**
- 長く生存しているほど、受けるおじゃま量が微減 or 相殺効率が微増。攻撃が下手でも「耐える」戦法に勝ち筋を与える。

> **調整上の注意**：ラバーバンドを強くしすぎると上手い人が報われずストレスになる。スキルが勝率に相関しつつ、勝敗を決定論的にはしない塩梅を探る（→ §11）。

## 4. 入力方式・課題

### 4.1 入力方式（確定：日本語ローマ字入力）
- **日本語ローマ字入力**で確定（寿司打 / e-typing 形式）。お題はかな表記で表示し、ローマ字打鍵を判定。
- **複数許容ローマ字**への対応が判定ロジックの肝。主要な揺れを許容する：
  - し＝shi/si、ち＝chi/ti、つ＝tsu/tu、ふ＝fu/hu、じ＝ji/zi、を＝wo
  - ん＝n/nn（実装簡易化のため nn を常時許容してよい）
  - 拗音：しゃ＝sha/sya、ちゃ＝cha/tya、じゃ＝ja/jya/zya、きゃ＝kya …
  - 促音っ＝次の子音を重ねる（例：きって＝kitte）。
- オプションで英単語/英文モードも想定（判定が単純で、先行プロトタイプ向き）。

### 4.2 速度指標
- 内部は **KPM（打鍵数/分）** を基準（日本語タイピングの標準）。
- 英語モードでは WPM 換算も表示可。

### 4.3 お題プール
- 全プレイヤー共通の**シード値**から決定論的に単語列を生成 → 初期難易度の公平性を確保。
- おじゃま挿入で各自の列は途中から分岐するため、厳密な同一進行にはこだわらない。
- ジャンル（短文/単語/カテゴリ選択）は将来拡張。

## 5. ゲームの流れ

1. **ロビー/ルーム**：プレイヤーが入室。ホストが開始 or 最小人数＋カウントダウンで自動開始。
2. **カウントダウン**：サーバタイムスタンプ基準の `startAt` を全員で共有し、同期スタート。
3. **対戦**：リアルタイム進行。攻撃・相殺・脱落が発生。
4. **脱落**：トップアウトしたプレイヤーは順位確定 → 観戦モードへ移行。
5. **決着**：生存1人で終了。
6. **結果画面**：順位・KO数・最高連鎖・最高KPMなどを表示。

## 6. 技術仕様

### 6.1 アーキテクチャ
- **フロント**：HTML/JS（または React ビルド）を GitHub Pages で配信。
- **リアルタイム同期**：Firebase **Realtime Database**（Firestoreより低レイテンシ・高頻度更新向き）。
- **識別**：Firebase Auth（匿名）または UUID 永続化（イントロクイズと同方式）。

### 6.2 データ構造（Realtime DB / JSONスキーマ案）

```
/rooms/{roomId}
  /meta
    status      : "waiting" | "countdown" | "playing" | "finished"
    startAt     : <serverTimestamp ms>   // 同期開始時刻
    seed        : <number>                // 共通お題シード
    maxPlayers  : 16
    hostUid     : <uid>
  /players/{uid}
    name    : string
    alive   : boolean
    backlog : number     // 現在のバックログ数（mini-board表示用・スロットリング書込）
    combo   : number     // 現在の連鎖数
    kpm     : number     // 表示用の打鍵速度
    badges  : number
    rank    : number     // 脱落時に確定
    lastSeen: <serverTimestamp>   // 切断検知
  /attacks/{targetUid}/{pushId}
    from   : <uid>
    amount : number
    at     : <serverTimestamp>
```

### 6.3 同期戦略（最重要）
- **キーストロークは同期しない**。お題は共通シードから各クライアントがローカル生成。
- 各クライアントは**自分の表示用サマリ**（backlog / combo / kpm / alive / badges）のみを **300〜500msスロットリング**で `/players/{uid}` に書込。
- 全員の mini-board 表示用に `/players` を購読（1人あたり数値数個の軽量データ）。
- **攻撃のみイベントとして即時 push**：`/attacks/{targetUid}` に書込 → 対象が購読し、予告ゲージへ反映 → 相殺/確定処理後にノードを削除（consume）。

### 6.4 切断処理
- `onDisconnect()` で `alive=false` または `lastSeen` 更新。
- 一定時間 `lastSeen` 更新が止まったプレイヤーを脱落扱い（ホストまたは各クライアントで判定）。

### 6.5 Firebase 無料枠の制約（要注意）
- **Sparkプラン：Realtime DB 同時接続 100 がハードリミット**。
  → 観戦者や複数ルームを考えると、1ルーム99人は現実的でない。
- **推奨スコープ**：まず **1ルーム 8〜16人**で成立させる。「100人規模」は Blaze プラン＋複数DB shard 等で将来検討。
- 書込試算：16人 × 2回/秒 × 120秒 ≈ 3,840書込/試合 → 無料枠で十分余裕。

### 6.6 チート対策
- v0.1 はクライアント主導（client-authoritative）。カジュアル対戦前提なら許容。
- 将来：Cloud Functions でのサーバ検証、非現実的な打鍵速度の検知などを追加余地として確保。

## 7. 開発ロードマップ（フェーズ分け）

- **Phase 0 — ソロ基盤（オフラインで完結）**
  お題表示・ローマ字判定・連鎖・ベース供給＆加速・トップアウト。まずここを楽しく成立させる。
- **Phase 1 — オンライン接続**
  Firebase接続、ロビー/ルーム、`startAt`同期スタート、mini-board一覧表示。
- **Phase 2 — 攻撃送受信**
  おじゃま送信／受信、予告ゲージ、相殺ロジック。
- **Phase 3 — 戦略要素**
  ターゲティング4モード、バッジ／KOボーナス。
- **Phase 4 — 決着まわり**
  脱落・順位・観戦モード・結果画面・切断処理。
- **Phase 5 — 調整・演出・スケール検証**
  数値バランス調整、エフェクト、人数上限の検証。

## 8. 未決事項・要調整パラメータ

**要決定**
- お題ジャンル：短文 / 単語 / カテゴリ選択。
- 最大人数：8 / 16 / それ以上（暫定＝16）。
- アイテムの種類・効果・出現率、ラバーバンド補正の強さ。

**要プレイテスト調整（数値）**
- 連鎖マイルストーンの閾値とおじゃま量。
- ベース供給間隔と加速カーブ。
- バックログ上限（トップアウト閾値、暫定＝12〜15単語）。
- 予告ゲージの遅延時間（暫定＝1.5秒）。
- 速度ボーナスのKPM閾値、難単語の文字数閾値。
- バッジ倍率の上限。
